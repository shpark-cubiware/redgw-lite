RedGW Overview 개요
RedGW(Redis Gateway)는 이기종 시스템 간 데이터 공유를 위한 REST 기반 인메모리 데이터 게이트웨이입니다.
인사관리(HRM), 전사자원관리(ERP), 영업·고객관리(CRM) 등 서로 다른 업무 시스템이 표준화된 REST API를 통해 실시간으로 데이터를 공유할 수 있습니다.
핵심 가치: 이기종 시스템 간 메모리에 적재된 데이터의 빠른 공유와 표준화된 공유 방법 제공

비영속 원칙 (Ephemeral)

중요: Redis는 순수 인메모리로 운영됩니다. RDB/AOF 모두 비활성화되어 있으며, 서비스 중단/재시작 시 모든 데이터가 소멸됩니다. 영구 저장이 필요한 데이터는 각 시스템의 자체 DB에 보관해야 합니다.

타입 접두어가 필요한 이유
Redis에서 한번 만들어진 키는 고유한 데이터 타입을 가집니다. 예를 들어 ERP:status를 String으로 저장한 뒤, 같은 키에 Hash 명령을 실행하면 WRONGTYPE 에러가 발생합니다.

타입 접두어를 붙이면 이 문제를 원천 차단할 수 있습니다.
  • 충돌 방지 — ERP:kv:status(String)와 ERP:map:status(Hash)는 서로 다른 키이므로 공존 가능
  • 타입 식별 — 키 이름만 보고 데이터 타입을 즉시 파악 가능
  • 타입별 조회 — KEYS ERP:kv:*로 String 키만 필터링 가능
Data Type Selection Guide 타입 선택
RedGW는 6가지 Redis 데이터 타입을 REST API로 제공합니다. 데이터의 특성에 따라 적절한 타입을 선택하세요.
타입Redis 원형적합한 상황실전 예시
KV (String) String 단순 값, 캐시, 플래그, 카운터, 분산 락 시스템 상태값 일련번호 생성 작업 락
Map (Hash) Hash 구조화된 레코드, 다중 필드 객체 주문 정보 사용자 프로필 배정 결과
Queue (List) List 순서 있는 작업 대기열, FIFO 처리 배정 요청 큐 알림 대기열 로그 버퍼
Group (Set) Set 고유 멤버 집합, 존재 확인, 집합 연산 온라인 사용자 처리 완료 목록 태그
Rank (Sorted Set) Sorted Set 점수 기반 정렬, 우선순위, 랭킹 주문 우선순위 가치 랭킹 배정 스코어
Event (Stream) Stream 시간순 이벤트 로그, Consumer Group 주문 등록 이벤트 감사 로그 변경 이력
Pub/Sub Pub/Sub 실시간 브로드캐스트 (저장 안됨) 긴급 알림 상태 변경 통보 실시간 채팅

API Key Setup 설정 안내
RedGW의 모든 API(Health 제외)는 X-API-Key 헤더를 통한 인증이 필요합니다. API Key는 관리자가 config/config.yaml에 등록하며, 운영 시에는 환경변수로 오버라이드합니다.

1. API Key 발급

API Key는 관리자에게 발급받습니다.
설정 파일(config/config.yaml)의 clients 항목에 등록되며, 환경변수 REDGW_CLIENT_{ID}_API_KEY로 오버라이드됩니다.
예: REDGW_CLIENT_ERP_API_KEY=your-actual-key-here

2. 이 페이지에서 API Key 설정

컨테이너 시작 시 .env의 키 값을 /demo/env.js에 주입하여 페이지 로드 때 자동 등록합니다. 정상 로드 시 상단 API Key 입력란이 자동으로 채워집니다.
상황대처
정상페이지 로드 시 .env의 키가 자동으로 입력란에 표시됨
CHANGE_ME 상태.env에 실제 키를 입력 후 컨테이너 재시작
env.js 로드 실패상단 API Key 입력란에 직접 키 입력 (브라우저에 자동 저장됨)
자동 저장: 수동으로 입력한 키는 브라우저의 localStorage에 클라이언트별로 저장되며, 페이지를 새로고침하거나 재방문해도 유지됩니다. env.js 로드 키는 컨테이너 재시작마다 갱신됩니다.

3. API 호출 시 인증

# 모든 API 요청에 X-API-Key 헤더를 포함 curl -k -s -X GET "https://host:3443/api/v1/ns/ERP/kv/status" \ -H "X-API-Key: your-api-key-here" # WebSocket은 쿼리 파라미터로 전달 (HTTPS 환경에서는 wss://) wss://host:3443/api/v1/ws/ERP/subscribe/alert?api_key=your-api-key-here

4. curl 호출 시 API Key 설정

매번 긴 API Key를 입력하지 않도록 환경변수로 등록해 두면 편리합니다. .env 파일의 값을 사용하세요.

Patterns & Recipes 활용 패턴
RedGW의 데이터 타입을 조합하여 구현할 수 있는 실전 활용 패턴입니다. 각 패턴은 실무에서 자주 사용되는 분산 시스템 설계 기법입니다.

1. 분산 락 (Distributed Lock)

동시에 하나의 프로세스만 특정 주문을 편집하도록 보장합니다. PUT /kv/{key}/nx(SETNX)로 락을 획득하고, TTL(1시간)로 자동 해제를 보장합니다. key=락 대상(주문번호) · value=락 소유자(해제 시 본인 확인용) · ttl=자동 해제 시간.
Step 1 [락 획득] PUT /ns/{ns}/kv/lock:order-2026-000123/nx body: {"value": "worker-hrm-01", "ttl": 3600} → 200 (성공) / 409 (이미 잡힘) ↓ Step 2 [작업 수행] 주문 편집, 보고서 생성 등 ↓ Step 3 [락 해제] DELETE /ns/{ns}/kv/lock:order-2026-000123 ※ TTL(1시간)이 있으므로 비정상 종료 시에도 자동 해제 (데드락 방지)

2. Producer-Consumer (작업 큐)

생산자가 작업을 큐에 등록하면, 소비자가 순서대로 꺼내서 처리합니다. Queue의 FIFO(First In, First Out) 특성을 활용합니다.
Step 1 [Producer — 작업 등록] POST /ns/ERP/queue/work-queue body: {"value": "{\"task\":\"분석\",\"id\":\"001\"}", "direction": "right"} → 오른쪽으로 push (RPUSH) ↓ Step 2 [Consumer — 작업 소비] GET /ns/ERP/queue/work-queue/pop?direction=left → 왼쪽에서 pop (LPOP) — FIFO 순서 보장 ↓ Step 3 [모니터링] GET /ns/ERP/queue/work-queue/len → 대기 중인 작업 수 확인

3. Event Sourcing (이벤트 소싱)

모든 변경을 이벤트로 기록하고, Consumer Group을 통해 여러 워커가 분산 처리합니다. 이벤트는 시간순으로 보존되어 감사 로그로도 활용됩니다.
Step 1 [이벤트 발행] POST /ns/ERP/event/order-registered body: {"data": {"order_id": "2026-001", "type": "일반주문"}} ↓ Step 2 [컨슈머 그룹 생성] POST /ns/ERP/event/order-registered/group body: {"group": "my-worker-group"} ↓ Step 3 [워커 읽기 — 분산 소비] GET /ns/ERP/event/order-registered/group/my-worker-group/read ?consumer=worker-1 → 각 워커가 고유한 이벤트를 받음 (중복 없음) ↓ Step 4 [처리 완료 — ACK] POST /ns/ERP/event/order-registered/group/my-worker-group/ack body: {"ids": ["1234567890-0"]}

4. 실시간 알림 (Real-time Notification)

Pub/Sub을 사용하여 실시간으로 이벤트를 브로드캐스트합니다. WebSocket으로 구독하면 즉시 메시지를 수신합니다. 메시지는 저장되지 않습니다.
Step 1 [구독자 — WebSocket 연결] WS wss://host:3443/api/v1/ws/ERP/subscribe/alert → WebSocket 연결 유지, 메시지 대기 ↓ Step 2 [발행자 — 메시지 전송] POST /ns/ERP/publish/alert body: {"message": "주문 2026-001 긴급 업데이트"} → 모든 구독자에게 즉시 전달
Event vs Pub/Sub: 이벤트 보존이 필요하면 Event (Stream), 실시간 전달만 필요하면 Pub/Sub을 사용하세요.

5. 공유 시퀀스 생성기 (Shared Sequence)

여러 시스템이 공유하는 고유 일련번호를 원자적(atomic)으로 생성합니다. shared 네임스페이스의 INCR을 활용합니다.
POST /ns/shared/kv/seq:order-number/incr [시스템 A] 요청 → {"value": 1001} [시스템 B] 요청 → {"value": 1002} [시스템 C] 요청 → {"value": 1003} → 원자적(atomic) 증가, 중복 없음, 락 불필요 → shared NS이므로 모든 시스템이 read+write 가능

6. 교차 시스템 데이터 공유 (Cross-NS Sharing)

한 시스템이 저장한 데이터를 다른 시스템이 읽습니다. 권한 모델에 따라 교차 네임스페이스 읽기가 허용됩니다.
Step 1 [HRM — 데이터 저장] PUT /ns/HRM/map/assign:001 body: {"fields": {"score": "95", "staff": "kim-001"}} ← 자기 NS에 write ↓ Step 2 [ERP — 교차 조회 성공] GET /ns/HRM/map/assign:001 → ERP는 HRM에 대해 read 권한 있음 → 200 OK ↓ Step 3 [CRM — 교차 조회 거부] GET /ns/HRM/map/assign:001 → CRM는 HRM read 권한 없음 → 403 NAMESPACE_DENIED
